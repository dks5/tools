Защита Статических Страниц

Детализированное руководство по клиентской парольной защите

Обзор Метода

**GitHub Pages** не поддерживает серверную аутентификацию. Чтобы обойти это ограничение, мы будем использовать инструмент **StatiCrypt**, который запускается **на вашем личном компьютере** (не на сервере) и шифрует ваш HTML-файл с помощью алгоритма **AES-256**.

В результате вы получите готовый, зашифрованный HTML-файл. Когда пользователь откроет его, он увидит только форму для ввода пароля. Расшифровка произойдет мгновенно в его браузере, и сервер GitHub Pages в этом процессе не участвует.

⚠️ Важно: Уточнение по серверу

Вам не нужен веб-сервер для этого процесса. **Node.js** и **npm** — это инструменты, которые вы установите на свой компьютер (Windows, macOS или Linux), чтобы запустить программу **StatiCrypt** один раз для шифрования файла.

⭐ **Параметры:** `-p` устанавливает пароль; `-o` задает имя выходного файла. **Обратите внимание:** команда разбита на строки для удобства чтения.

Это критически важно! Убедитесь, что оригинальный, незашифрованный файл удален, чтобы он не был доступен по прямому URL.

Вопросы и Ответы (FAQ)

Вопрос: Файл шифруется один раз, и этот файл будет использован постоянно? Доступ к странице будет через пароль с любого браузера?

1. Частота шифрования (Один раз или повторно?)

Вы абсолютно правы: **шифрование выполняется только один раз** для создания защищенной версии файла. Программа **StatiCrypt** работает как утилита, которая берёт ваш чистый HTML-файл и генерирует на его основе новый, зашифрованный файл, содержащий форму для пароля и код дешифровки.

  • Если контент не меняется: Вы загружаете зашифрованный файл (`protected-server-setup.html`) на GitHub Pages и можете использовать его **неограниченное время**. Повторно запускать StatiCrypt не нужно.
  • Если вы меняете контент (обновление): Вы вносите правки в исходный (незашифрованный) HTML-файл на своём компьютере, а затем **обязательно повторяете Шаг 3** с командой `staticrypt`. Это создаст новый зашифрованный файл с обновлённым содержимым, который затем заменит старый файл на GitHub (Шаг 4).

2. Доступность (С любого браузера и из любого места?)

Да, это ключевое преимущество статического шифрования: **доступ возможен из любого браузера, с любого устройства и из любой точки мира**. Это происходит потому, что:

  1. **GitHub Pages** просто отдаёт HTML-файл.
  2. **Весь процесс расшифровки** (проверка пароля, дешифровка контента AES-256) происходит локально **в браузере пользователя** (на стороне клиента).
  3. Никакие секретные данные или пароли не отправляются обратно на сервер GitHub.

Ваша страница всегда будет работать по принципу: "файл загружен -> введён пароль -> контент разблокирован".

Вопрос: А где хранится оригинальный файл HTML?

Безопасное хранение оригинального файла

Это критически важный вопрос безопасности. **Оригинальный HTML-файл (`server-setup-duckdns-ssl-v1.html`) должен храниться ТОЛЬКО на вашем локальном компьютере, в безопасном месте, и НИКОГДА не должен быть загружен на GitHub Pages.**

🔥 Обязательное требование безопасности (Шаг 4):

Вы должны **удалить** оригинальный, незащищенный файл из публичного репозитория GitHub. Если он останется, злоумышленник сможет обойти пароль, просто зная прямой URL к этому незащищенному файлу.

Роли двух файлов:

  • **Оригинал (Локально):** Вы используете его как мастер-копию для внесения правок и обновлений. Он не предназначен для публичного доступа.
  • **Защищенный файл (На GitHub Pages):** Это результат работы StatiCrypt. Он содержит только зашифрованный текст и скрипт разблокировки. Это единственный файл, который должен быть в публичном доступе.

Помните: ваш зашифрованный файл — это как сейф. Если вы оставите ключ от сейфа (оригинальный файл) рядом, вся защита теряет смысл.

Вопрос: Могу ли я создать приватный репозиторий для хранения оригинальных HTML-файлов? Насколько это надежная защита?

Краткий ответ: Да, это отличная и рекомендуемая стратегия.

**Хранение оригиналов в приватном репозитории GitHub** — это абсолютно достаточное и, более того, рекомендуемое решение для обеспечения безопасности ваших исходных, незашифрованных HTML-файлов. Благодаря тому, что GitHub позволяет создавать приватные репозитории бесплатно, вы можете надежно скрыть исходный код от публичного доступа.

Обеспечение высокого уровня защиты:

  • Разделение доступов (Ключевой момент): Вы четко разделяете публичный контент (только зашифрованный файл) и исходники. Исходник (`server-setup-duckdns-ssl-v1.html`) надежно скрыт за механизмом аутентификации GitHub.
  • Защита аккаунта GitHub: Файлы защищены вашим логином, паролем и, что критически важно, **двухфакторной аутентификацией (2FA)**. До тех пор, пока ваш аккаунт GitHub не взломан, ваши исходники в полной безопасности.
  • Соответствие рабочему процессу: Это решение полностью соответствует критическому требованию безопасности из Шага 4 инструкции: оригинальный, незашифрованный файл удален из публичной части, и его прямой URL недоступен.

⭐ Рекомендации для максимальной надежности:

  • **Обязательно включите двухфакторную аутентификацию (2FA)** для вашего аккаунта GitHub.
  • Используйте **очень длинный и уникальный пароль** для шифрования публичного файла StatiCrypt (параметр `-p` в Шаге 3).

Приватный репозиторий — идеальное место для "мастер-копии", которую вы редактируете, прежде чем повторно зашифровать и заменить публичную версию.

Углубленная очистка (Cleanup)

После завершения работы с Node.js и StatiCrypt вы можете удалить их и очистить системные переменные, чтобы освободить место и избежать конфликтов.

Углубленная очистка macOS: Удаление Homebrew, Node.js и оптимизация PATH

**Внимание:** Убедитесь, что вам действительно не нужны эти программы, так как их удаление необратимо.

1. Полное удаление Homebrew (Brew)

Homebrew — это пакетный менеджер, который может занимать значительное место, включая кэши и установленные им программы.

1.1 Выполнение скрипта деинсталляции Homebrew

Используйте официальный скрипт для удаления Homebrew и всех его компонентов.

  1. Откройте **Терминал**.
  2. Выполните следующую команду (скопируйте целиком): 
    /bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/uninstall.sh)"
  3. Скрипт спросит, что именно вы хотите удалить (только Homebrew или также все установленные им пакеты). Для максимальной очистки выберите **удаление всего**.

1.2 Проверка и удаление оставшихся папок

После выполнения скрипта вручную проверьте, не остались ли папки, которые мог создать Homebrew.

# Проверка, остались ли папки Homebrew
ls -ld /usr/local/Homebrew
ls -ld /opt/homebrew # Если у вас Mac с чипом Apple Silicon (M1/M2/M3)

Если команды вернут ошибку "No such file or directory" (Нет такого файла или каталога), значит, папки удалены. Если папки существуют, удалите их (будьте очень осторожны с командой `rm -rf`!):

# Удаление на Intel Mac (если осталось)
sudo rm -rf /usr/local/Homebrew

# Удаление на Apple Silicon Mac (если осталось)
sudo rm -rf /opt/homebrew

2. Полное удаление Node.js и npm

2.1 Удаление Node.js, npm и глобальных модулей

Эта команда удаляет основные исполняемые файлы Node.js, npm, и каталоги глобальных модулей.

# 1. Удаление глобальных пакетов npm (может потребовать ручного подтверждения)
sudo npm uninstall -g $(ls -1 /usr/local/lib/node_modules/)

# 2. Удаление основных папок Node.js 
sudo rm -rf /usr/local/{bin/{node,npm},lib/node_modules,lib/dtrace/node,share/man/man1/{node.1,npm.*}}

# 3. Удаление глобального кэша npm и конфигурационных файлов
rm -rf ~/.npm
rm -rf ~/.nvm # Если вы использовали NVM (Node Version Manager)

После этих действий Node.js и npm должны быть полностью удалены из системы.

3. Очистка переменной среды PATH

После удаления Brew и Node.js в ваших конфигурационных файлах могут остаться ссылки на несуществующие пути. Удаление этих ссылок ускорит запуск Терминала.

3.1 Идентификация конфигурационного файла оболочки

Определите, какую оболочку (shell) вы используете (Zsh или Bash). Выполните команду:

echo $SHELL

Если вывод `/bin/zsh`, то вам нужен файл `~/.zshrc`. Если `/bin/bash`, то `~/.bash_profile`.

3.2 Редактирование и очистка PATH

  1. Откройте соответствующий файл для редактирования (например, `nano ~/.zshrc`): 
    nano ~/.zshrc
  2. Найдите и **удалите** все строки, которые содержат упоминание:
    • Homebrew (например, `export PATH="/usr/local/bin:$PATH"`)
    • Node.js или npm
    • NVM (например, строки, начинающиеся с `export NVM_DIR=...`)
  3. После редактирования нажмите `Ctrl+O` (для сохранения), затем `Enter`, и `Ctrl+X` (для выхода из редактора `nano`).
  4. Перезагрузите конфигурацию, чтобы изменения вступили в силу: 
    source ~/.zshrc # или source ~/.bash_profile
  5. **Проверка:** Убедитесь, что удаленные пути больше не отображаются в `PATH`: 
    echo $PATH

4. Удаление неиспользуемых переменных окружения

Просмотрите список всех переменных и вручную удалите те, которые больше не нужны.

  1. Просмотрите список всех переменных: 
    env
  2. Снова откройте свой конфигурационный файл (`.zshrc` или `.bash_profile`) и **удалите** строки, которые устанавливают ненужные переменные (они обычно начинаются с `export ИМЯ_ПЕРЕМЕННОЙ=...`).
  3. Сохраните файл и выполните команду `source` для его перезагрузки.

Углубленная очистка Windows: Удаление Node.js и очистка PATH

**Внимание:** Убедитесь, что вам действительно не нужны эти программы, так как их удаление необратимо.

1. Деинсталляция Node.js через Программы и компоненты

1.1 Использование стандартного установщика

  1. Нажмите `Win + R` и введите `appwiz.cpl` для открытия "Программы и компоненты" (или "Установка и удаление программ").
  2. Найдите в списке **Node.js**.
  3. Щелкните правой кнопкой мыши и выберите **"Удалить"**.
  4. *Примечание:* Это удалит исполняемые файлы, но может оставить глобальные пакеты и кэш.

2. Удаление глобальных пакетов и кэша npm

Очистка этих папок вручную гарантирует максимальное освобождение места.

2.1 Очистка папок вручную

  1. **Удаление глобального кэша npm:**

    Откройте Проводник и перейдите в папку: C:\Users\<Ваше_Имя_Пользователя>\AppData\Roaming\npm-cache

    Удалите **содержимое** папки `npm-cache`.

  2. **Удаление глобальных модулей npm:**

    Откройте Проводник и перейдите в папку: C:\Users\<Ваше_Имя_Пользователя>\AppData\Roaming\npm

    Удалите **содержимое** папки `npm`.

3. Проверка и очистка переменной среды PATH

После удаления Node.js необходимо проверить и удалить все оставшиеся ссылки в `PATH`.

3.1 Открытие редактора переменных среды

  1. Нажмите `Win + R` и введите `sysdm.cpl`.
  2. Перейдите на вкладку **"Дополнительно"**.
  3. Нажмите кнопку **"Переменные среды..."** внизу.

3.2 Редактирование системной переменной PATH

  1. В окне "Переменные среды" в разделе **"Системные переменные"** найдите переменную **`Path`** и выделите ее.
  2. Нажмите **"Изменить..."**.
  3. В открывшемся списке найдите и **удалите** все записи, явно связанные с Node.js или npm. Обычно это:
    • `C:\Program Files\nodejs\`
    • `C:\Users\<Ваше_Имя_Пользователя>\AppData\Roaming\npm` (проверьте также "Пользовательские переменные")
  4. Нажмите **ОК** во всех окнах, чтобы сохранить изменения.

3.3 Проверка изменений

Откройте **новую** Командную строку (Command Prompt) или PowerShell и выполните команды:

node -v
npm -v
echo %PATH%

Команды `node -v` и `npm -v` должны выдать ошибку ("команда не найдена"), а в выводе `echo %PATH%` не должно быть ссылок на папки Node.js.
